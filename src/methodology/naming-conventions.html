Naming Conventions

Consistent naming patterns for files, folders, classes, and variables across the project.

File Naming

SCSS Files

  • Start with an underscore: _component-name.scss
  • Use lowercase, kebab-case
  • One component per file

Examples:

  • _buttons.scss
  • _cards.scss
  • _search-box.scss
  • _form-inputs.scss

HTML Files

  • Use lowercase, kebab-case
  • Be descriptive
  • Use hyphens to separate words

Examples:

  • demo-buttons.html (component demo)
  • guides-in-accordions.html (mockup page)
  • design-principles.html (documentation)

Folder Names

  • Use lowercase
  • Use hyphens for multi-word folders
  • Keep names short and descriptive

Examples:

  • design_system/
  • design_mockups/
  • methodology/

CSS Class Naming: BEM (Block–Element–Modifier)

BEM Pattern: .block__element—modifier

When to Use BEM

Apply BEM judiciously based on component complexity:

Type Use BEM? Example
Layout rules
(columns, grids)		 ❌ No Use Bootstrap utilities: col-md-6, d-flex
Bespoke components
(search box, cards, nav)		 ✅ Yes .search-box__input
.card__header—primary
One-off elements
(custom spacers)		 ❌ No Use Bootstrap utilities: mt-4, p-3

BEM Rules

  • Block: Top-level component (e.g., .card, .search-box)
  • Element: Child of a block, preceded by __ (e.g., .card__header, .search-box__input)
  • Modifier: Variation of a block or element, preceded by — (e.g., .card—featured, .btn—primary)

Good BEM Examples

// News Card Component
.news-card { }
.news-card__image { }
.news-card__content { }
.news-card__title { }
.news-card__description { }
.news-card—featured { }
.news-card—dark { }

// Search Widget
.search-widget { }
.search-widget__input { }
.search-widget__button { }
.search-widget__suggestions { }
.search-widget—expanded { }

What NOT to Do

  • ❌ Don't over-nest: .card__content__inner__text is too deep
  • ❌ Don't mix BEM with Bootstrap utilities randomly
  • ❌ Don't use BEM for simple spacing or alignment
  • ❌ Don't create modifier classes without a related block: just use util classes

SCSS Variable Naming

Variables are organized in three levels of abstraction:

Level 1: Core Brand/Utility (Foundational)

$primary: #800000;       // Brand maroon
$secondary: #737373;     // Dark greystone
$success: #789D4A;       // Ivy green
$danger: #A4343A;        // Brick red

Level 2: Semantic Generic Roles (Reusable)

$background: #ffffff;
$text-primary: #000000;
$text-muted: #737373;
$border-subtle: #D9D9D9;

Level 3: Component-Specific Roles

$btn-primary-bg: $primary;
$btn-primary-text: white;
$search-placeholder-text: $text-muted;
$card-border: $border-subtle;

General Variable Rules

  • Use kebab-case (e.g., $text-muted, not $textMuted)
  • Be descriptive (e.g., $spacing-sm instead of $gap1)
  • Group related variables with comments
  • Define all variables in _variables.scss or at the top of component files

Avatar & Icon Naming

  • Font Awesome Icons: Use the standard Font Awesome class names 
    <i class="fa-solid fa-magnifying-glass"></i>
  • Custom Icon Classes: Follow BEM for icon components 
    .icon-button__icon { }
.icon-button__label { }

ID Naming

  • Use IDs sparingly—prefer classes for styling
  • Reserve IDs for unique page elements and form labels
  • Use kebab-case: id="main-nav", not id="mainNav"
  • Be descriptive: id="search-form", not id="form1"
